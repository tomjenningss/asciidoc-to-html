<div class="sect1">
<h2 id="SSAA">Using Liberty Social Login with Red Hat OpenShift</h2>
<div class="sectionbody">
<div class="paragraph">
<p>The Social Login feature <code>socialLogin-1.0</code> can now be configured to use OpenShift’s built-in OAuth server and OAuth Proxy sidecar as authentication providers. The Social Login feature has several pre-configured providers (e.g. Google, GitHub, Facebook) but you can also configure additional providers (e.g. Instagram). OpenShift&#8217;s OAuth server and OAuth Proxy sidecar can now be configured as additional providers too. The first is a standard OAuth Authorization Code flow, where a web browser accessing an app running in Liberty is redirected to the OpenShift OAuth server to authenticate.
The second is accepting an inbound token from the OpenShift OAuth Proxy sidecar or obtained from an OpenShift API call. This approach requires less cluster-specific configuration.</p>
</div>
<div class="paragraph">
<p>Most users of this will run Liberty in a pod; however, in the Authorization Code flow, Liberty can run outside the OpenShift cluster. In either mode, an optional JWT can be created for propagation to downstream services.</p>
</div>
<div class="paragraph">
<p>Using OpenShift as a provider differs slightly from other OAuth providers, it requires a service account token to obtain information about the OAuth tokens.
Once the: client ID, secret, and token have been obtained from OpenShift, Liberty can be configured as shown below.</p>
</div>
<div class="paragraph">
<p>To enable the feature add it to the <code>server.xml</code></p>
</div>
<div class="paragraph">
<p>Server configuration to use OpenShift OAuth server:</p>
</div>
<div class="listingblock">
<div class="content">
<pre class="highlight"><code class="language-xml" data-lang="xml">&lt;server description="social"&gt;

  &lt;!-- Enable features --&gt;
  &lt;featureManager&gt;
    &lt;feature&gt;appSecurity-3.0&lt;/feature&gt;
    &lt;feature&gt;socialLogin-1.0&lt;/feature&gt;
  &lt;/featureManager&gt;

&lt;logging traceSpecification="com.ibm.ws.security.*=all=enabled" maxFiles="8" maxFileSize="200"/&gt;

&lt;httpEndpoint  id="defaultHttpEndpoint" host="*" httpPort="8941" httpsPort="8946" &gt; &lt;tcpOptions soReuseAddr="true" /&gt; &lt;/httpEndpoint&gt;


  &lt;!-- specify your clientId, clientSecret and userApiToken as liberty variables or environment variables --&gt;
  &lt;oauth2Login id="openshiftLogin"
    scope="user:full"
    clientId="${myclientId}"
    clientSecret="${myclientSecret}"
    authorizationEndpoint="https://oauth-openshift.apps.papains.os.fyre.ibm.com/oauth/authorize"
    tokenEndpoint="https://oauth-openshift.apps.papains.os.fyre.ibm.com/oauth/token"
    userNameAttribute="username"
    groupNameAttribute="groups"
    userApiToken="${serviceAccountToken}"
    userApiType="kube"
    userApi="https://api.papains.os.fyre.ibm.com:6443/apis/authentication.k8s.io/v1/tokenreviews"&gt;
  &lt;/oauth2Login&gt;

  &lt;keyStore id="defaultKeyStore" password="keyspass" /&gt;

  &lt;!-- more application config would go here --&gt;

&lt;/server&gt;</code></pre>
</div>
</div>
<div class="paragraph">
<p>In the sidecar scenario, the configuration changes to accept an inbound token from the sidecar.
Server configuration to use OAuth proxy sidecar:</p>
</div>
<div class="listingblock">
<div class="content">
<pre class="highlight"><code class="language-xml" data-lang="xml">&lt;!-- specify your userApiToken as a liberty variable or environment variable --&gt;
  &lt;!-- note that no clientId or clientSecret are needed --&gt;
  &lt;oauth2Login id="openshiftLogin"
    scope="user:full"
    userNameAttribute="username"
    groupNameAttribute="groups"
    userApiToken="${serviceAccountToken}"
    userApiType="kube"
    accessTokenHeaderName="X-Forwarded-Access-Token"
    accessTokenRequired="true"
    userApi="https://kubernetes.default.svc/apis/authentication.k8s.io/v1/tokenreviews"&gt;
  &lt;/oauth2Login&gt;</code></pre>
</div>
</div>
<div class="paragraph">
<p>Use of HTTPS communication requires that the server either have a key signed by a well-known certificate authority, which Liberty can trust automatically or that the server&#8217;s public key be added to the Liberty trust store. OpenShift does not come with CA-signed keys by default, so the Red Hat OpenShift OAuth server&#8217;s public key will need to be added. The most convenient way to do this is to specify an environment variable in <code>server.env</code>. That identifies the file containing the public key in PEM format. Liberty will read the file and add the key to its trust store.</p>
</div>
<div class="listingblock">
<div class="content">
<pre class="highlight"><code class="language-xml" data-lang="xml"># server.env

# OAuth sidecar scenario: causes the Kubernetes default certificate that is pre-installed in pods to be added to Liberty trust store.
cert_defaultKeyStore=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# OAuth server scenario: causes the public keys from /tmp/trustedcert.pem (obtained seperetly) to be added to Liberty trust store.
cert_defaultKeyStore=/tmp/trustedcert.pem</code></pre>
</div>
</div>

<div class="sect1">
<h2 id="NMM">Monitor the process CPU time (MicroProfile Metrics 2.0)</h2>
<div class="sectionbody">
<div class="paragraph">
<p>A new metric, <code>processCpuTime,</code> which returns the CPU time used by the process on which the JVM is running. The MicroProfile Metrics feature provides information monitoring an application, such as CPU time used, memory heap, response time of servlets.</p>
</div>
<div class="paragraph">
<p>The new <code>processCpuTime</code> metric provides a more accurate CPU load percentage on cloud platforms via Grafana. Previously, the CPU load percentage was shown with the metric <code>processCpuLoad</code>. However, the load percentage was calculated using the total number of cores allocated to the deployment. If the deployment has a restricted number of cores, the <code>processCpuLoad</code> ends up showing a plateau on Grafana when the maximum number of cores is reached. For example, on a deployment with 32 cores allocated but restricted to four cores, the <code>processCpuLoad</code> graph shows a plateau at 12.5% when all four cores are used.</p>
</div>
<div class="paragraph">
<p>The new metric, <code>processCpuTime,</code> can be manipulated on Grafana to create a more accurate representation of the CPU being used. <code>rate(processCpuTime)[1m]</code> shows the average rate of increase in CPU time over one minute. Dividing this by the total number of CPU cores, we can see a more accurate percentage of the CPU used, taking into account the constraints.</p>
</div>
<div class="paragraph">
<p>The new <code>processCpuTime</code> metric is displayed on the <code>/metrics</code> endpoint with the MicroProfile Metrics 2.0 and 2.2 features. On the dashboard, a new panel can be created with the following PromQL query: <code>(rate(base:cpu_process_cpu_time[2m])/1e9) / base:cpu_available_processors{app=~<a id="app"></a>}.</code> <a href="https://github.com/OpenLiberty/open-liberty-operator/tree/master/deploy/dashboards/metrics">View full dashboard.</a></p>
</div>
<div class="paragraph">
<p>The following images show that the old metric, <code>processCpuLoad</code>, plateaus at 12.5% (4/23), while the new metric, <code>processCpuTime</code>, more accurately represents the percentage of CPU used.</p>

<h2 id="_faster_application_startups_with_liberty_annotation_caching">Faster application startups with Liberty annotation caching</h2>
<div class="sectionbody">
<div class="paragraph">
<p>Application startup times have been made faster by adding cache capabilities <code>annotation caching</code> to core class and annotation scanning function. Depending on application characteristics, startup times are reduced by 10% to more than 50%. Applications with many jar files, or which use CDI or JAX-RS function, see the best improvements:</p>
</div>
<div class="imageblock text-center">
<div class="content">
<img src="/img/blog/190011-annotationcaching-graph.png" alt="190011 annotationcaching graph">
</div>
</div>
<div class="paragraph">
<p>Good news! Annotation caching is enabled by default.</p>
</div>
<div class="paragraph">
<p>Annotation cache data is stored in the server workarea. Cache of application class data is cleared when performing a clean server start (starting the server with the <code>--clean</code> option). In normal operations, the clearing of cache data is not necessary, since the cache automatically regenerates cache data for changed application classes.</p>
</div>
<div class="paragraph">
<p>In container environments, for annotation caching to be effective, the server image must be "warmed" when the container image is created. Warming the server an be done by starting and stopping the server during the container build. Warming the image moves the annotation scan into the container build meaning you get optimal startup on the container deployment. Using the configure.sh file in the base open-liberty docker images causes the server to be started and stopped during the container build.</p>
</div>
</div>

<h2 id="acr">Automatically compress HTTP responses</h2>
<div class="sectionbody">
<div class="paragraph">
<p>You can now try out HTTP response compression.</p>
</div>
<div class="paragraph">
<p>Previous to this feature, Liberty only considered compression through the use of the <code>$WSZIP</code> private header. There was no way for a customer to configure the compression of response messages. Support now mainly consists of using the <code>Accept-Encoding</code> header in conjunction with the <code>Content-Type header</code>, of determining if compression of the response message is possible and supported. It allows the Liberty server to compress response messages when possible. It is beneficial because customers will want to use the compression feature to help reduce network traffic, therefore reducing bandwidth and decreasing the exchange times between clients and Liberty servers.</p>
</div>
<div class="paragraph">
<p>A new element, <code>&lt;compression&gt;</code>, has been made available within the <code>&lt;httpEndpoint&gt;</code> for a user to be able to opt-in to using the compression support.</p>
</div>
<div class="paragraph">
<p>The optional <code>types</code> attribute will allow the user to configure a comma-delimited list of content types that should or should not be considered for compression. This list supports the use of the plus “+” and minus “-“ characters, to add or remove content types to and from the default list. Content types contain a type and a subtype separated by a slash “/“ character. A wild card "*" character can be used as the subtype to indicate all subtypes for a specific type.</p>
</div>
<div class="paragraph">
<p>The default value of the types optional attribute is: <code>text/*, application/javascript</code>.</p>
</div>
<div class="paragraph">
<p>Configuring the optional <code>serverPreferredAlgorithm</code> attribute, the configured value is verified against the “Accept-Encoding” header values. If the client accepts the configured value, this is set as the compression algorithm to use. If the client does not accept the configured value, or if the configured value is set to ‘none’, the client preferred compression algorithm is chosen by default.</p>
</div>
<div class="listingblock">
<div class="content">
<pre class="highlight"><code class="language-xml" data-lang="xml">&lt;httpEndpoint  id="defaultHttpEndpoint"
        httpPort="9080"
        httpsPort="9443"&gt;
    &lt;compression types=“+application/pdf, -text/html” serverPreferredAlgorithm=“gzip”/&gt;&lt;/httpEndpoint&gt;</code></pre>
</div>
</div>
<div class="paragraph">
<p>Open Liberty supports the following compression algorithms: <code>gzip</code>, <code>x-gzip</code>, <code>deflate</code>, <code>zlib</code>, and <code>identity (no compression)</code></p>
</div>
<div class="paragraph">
<p>The <code>Http Response Compression</code> functionality has been designed from the following  <a href="https://github.com/OpenLiberty/open-liberty/issues/7502">Open Liberty Epic: #7502</a>. The design is outlined within the Epic for more detailed reading. The basic flow of the design is shown in the below diagrams:</p>
</div>
<div class="imageblock text-center">
<div class="content">
<img src="/img/blog/20001-http-response-compression-diagram.png" alt="20001 http response compression diagram">
</div>

<h2 id="GraphQL">You are now free to use GraphQL with Open Liberty!</h2>
<div class="sectionbody">
<div class="paragraph">
<p>In our latest OpenLiberty development builds, users can now develop and deploy GraphQL applications.  GraphQL is a complement/alternative to REST that allows clients to fetch or modify remote data, but with fewer round-trips.  Liberty now supports the (still under development) MicroProfile GraphQL APIs (<a href="https://github.com/eclipse/microprofile-graphql">learn more</a>) that allow developers to create GraphQL apps using simple annotations - similar to how JAX-RS uses annotations to create a RESTful app.</p>
</div>
<div class="paragraph">
<p>Developing and deploying a GraphQL app is cinch - take a look at this <a href="https://github.com/OpenLiberty/sample-mp-graphql">sample</a> to get started with these powerful APIs!</p>
</div>
